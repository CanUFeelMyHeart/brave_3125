# Инструкция по работе с Git   
(дополнение, Git Hub)  
> -----------------------  


Git необходим нам для того, чтобы хранить различные версии проекта. Чтобы мы, или наши коллеги по команде, могли вернуться к какому-либо этапу/версии проекта, и внести корректировку, что-то подправить или добавить.  


Порядок действий, по созданию **репозитория**, и работы в нем.  

**1.** Во-первых, мы создаем папку (или выбираем какую-нибудь из существующих), которую определим как репозиторий, хранилище различных версий нашего проекта.  
После того как мы выбрали папку, мы должны открыть ее в программе, в которой будем работать. Например: *Visual Studio Code*. Наводим курсор мыши на папку, жмем правую кнопку мыши, и выбираем "открыть с помощью Code".  
Теперь нам необходимо открыть терминал, и про-инициализировать папку программно, сделать из нее репозиторий. В верхнем меню *Visual Code* находим Терминал, и выбираем Создать терминал. В нижней части программы, у нас открывается терминал.  


**2.** Теперь мы должны *про-инициализировать* данную папку, чтобы из нее **создать репозиторий**.  
Делаем это, командой:  
> **git init**  


**3.** После того как мы создали репозиторий, необходимо **создать файл**, через меню в верхнем левой углу. Рядом с названием нашего репозитория, находятся несколько иконок. Нажимаем создать файл, и набираем название нашего файла. Например: file1.md  


**4.** Теперь можно начинать работать с файлом.  
После любой работы с файлами, обязательно делаем сохранение, сочетанием двух клавиш: **ctrl + s**  


**5.** Чтобы проверить статус нашего файла, набираем команду: 
> **git status**  

И мы видим сообщение, что файл изменен, и необходимо добавить файл в отслеживание, с указанием комментария, что было измененено или создано.  


**6.** Для того чтобы сохранить и добавить файл в Git, включить в фиксацию Git-ом всех изменений в файле, набираем команду с указанием имени файла (filename):  
> **git add filename**  


**7.** И далее, добавляем комментарий с ключиком -m и указанием текста в кавычках, что именно было измененено в файле с которым работали, командой:  
> **git commit -m "create file"**

чтобы мы могли посмотреть журнал изменений/сохранений.  


**10.** Для того чтобы перейти к нужному сохранению файла, набираем команду:  
> git checkout  +хэш-код

хэш-код это длинный ряд цифр. Так мы переходим в необходимое нам сейчас сохранение, работаем в нем, а после, возвращаемся обратно командой:  
> git checkout master

Помните, при каждом изменении файла, необходимо сохранять все изменения:  
> Ctrl+S, на клаватуре  
- далее, сразу же, командой в терминале  
> git add .  
- и обязательно комментируем, какие работы нами были сделаны  
> git commit -m "наш комментарий"  


Теперь мы готовы к работе с ветками.  
Создание ветки, осуществляется командой:  
> git branch new_branch_name  
- где new_branch_name это название новой ветки.  
После создания новой ветки, вы можете перейти в новую ветку, и продолжить работать с файлом, производя в нем какие-либо изменения. Например добавляя описание программ, функционала.  
Но также, помните, что необходимо сохранять все изменения.  


После того, как созданы новая ветка, или ветки..  
Кстати, мы ведь не поговорили о том, а для чего создается несколько веток.  
Представьте, что существует команда разработчиков, у каждого есть свои задачи, но каждый разработчик, работает в команде, в рамках одного проекта. А это значит, что каждый из них, может работать с одним и тем же файлом. Они могут производить изменения в одном файле, но только в рамках своих задач.  
И ветки, позволяют им это сделать.  
У каждого, есть доступ, как к ветке дерева. И где-то, они потом объединятся.  
Но об этом чуть позже.  

Так вот.  
Мы уже можем создать ветку. 
А чтобы переключиться на новую ветку, используем команду:  
> git checkout new_branch_name  


И мы можем продолжать работать с этим файлом, который кем-то уже был создан.  
И помним, сохранять, обязательно.  


Кстати, о сохранении, еще раз проговорим о важности внесения комментария, в каждом сохранении.  
Чтобы каждый коллега по команде, понимал, где какии изменения были произведены, и чтобы не допустить грубых ошыбок, чтобы все работали в стиле "плечом к плечу", комментарии приципиально важны.  
Не пропускайте их.  


Чуть позже, мы проговорим о том, как выглядит процесс слияния веток, в единую, одну ветку.  
Сейчас об удалении.  
После того, как производится слияние веток, большинство веток, уже не представляют ценности. И их можно удалить. Производится это командой:  
> git branch -d 'branch_name'  
 
 где 'branch_name', это название ветки, которую мы хотим удалить.  


Ну что-ж, основные команды известны.  
Можно поработать по-взрослому.

Кстати, мы чуть не забыли поговорить то о таких важных командах, как:  
> git log --graph 

команда, позволяющая нам посмотреть, когда и где были произведены создания и слияния веток.    

> git merge branch_name  

команда, которой мы и производим то самое слияние веток.  
Для этого, нам необходимо перейти в основную ветку, и набрать данную команду.  
И содержимое ветки branch_name, соединится с содержимым нашей основной ветки.  

Ну а после слияния всех веток в одну, основную, можно произвести финальную корректировку, и сдать работу на проверку.

## Дополнение к Инструкции.  
## Работа с Git Hub.  
Git Hub это сервис, который позволяет интегрироваться с программой Git, и настроить удаленную работу с вашим репозиторием (или с другими репозиториями).  

В Гит Хаб (Git Hub), мы можем отправлять как данные со своего локального репозитория, так и скачивать данные с чьего либо открытого репозитория.  
Необходимые действия, при работе в с Гит Хаб:  
+ первоначально необходимо создать аккаунт, зарегистрироваться в системе контроля версий Гит Хаб;  
+ после этого мы можем уже создать репозиторий, нажав в правом верхнем углу плюсик, выбрать __"новый репозиторий"__, дать ему имя, и нажать кнопку __"создать репозиторий"__;
+ в открывшемся меню, отображаются варианты того, как мы можем начать работы с репозиторием; создать новый, привязать к этому уже существвующий репозиторий, или импортировать код из другого репозитория; 
+ чтобы импортировать чей-то репозиторий, мы создаем папку, в которую положим репозиторий, идем в тот репозиторий на гит-хибе, который нам интересен, и делаем __fork__ (форк). Этим самым, мы копируем к себе на гит-хаб, ту интересную для нас ветку с которой мы сделали форк.  
+ Теперь со своего гит-хаба, мы делаем __Git clone__ этой интересной ветки, в папку, которую мы определили как будующий репозиторий для "интересной ветки", и можем теперь работать с данными этой ветки в Visual Studio Code (VSC).  
Только если мы захотим сделать push обратно, разработчику который публиковал данную ветку, нам необходимо будет делать пуш с отдельной ветки, не мастер и не мейн. 

+ при первом взаимодействии с новым репозиторием на Git Hub, необходимо авторизоваться И "подружить" локальный репозиторий с удаленным, куда будем отправлять наш код с локального репозитория (программа VS Code, помогает сама это сделать, будут подсказки, какие команды выбрать); и после того как мы это сделали, сможем увидеть в удаленном репозитории на Git Hub наш файл с текстом который создали в локальном;  
+ теперь с этим файлом могут работать все, кто имеет доступ к удаленному репозиторию; по ссылке в кнопке CODE, каждый может себе скачать себе данные тексты, файлы, командой __git clone__, и посмотреть у себя в локальномм репозитории;  
+ как и мы, зайдя в репозиторий который нам интересен и необходим, мы нажимаем кнопку CODE, копируем адрес https..., заходим в свой терминал, набираем команду __git clone__ и указываем тот самый скопированный адрес, нажимаем кнопку enter, и внутри в указанной нами папке, появляется новая папка с именем удаленного репозитория, и с содержимым удаленного репозитория; мы ОБЯЗАТЕЛЬНО переходим в эту папку (репозиторий), и локально можем  работать в нем; 
+ обязательно помните, когда мы клонируем удаленный репозиторий, себе на локальную машину, вызывать команду __git init__ не нужно, т.к. мы скопировали уже ГОТОВЫЙ репозиторий.
